## 2. Найденные медленные запросы (до оптимизации)

### Запрос №1: Поиск заказов пользователя за последний месяц

```sql
SELECT id, status, total_amount, created_at
FROM orders
WHERE user_id = (SELECT id FROM users WHERE email = 'user00001@example.com')
  AND created_at >= NOW() - INTERVAL '1 month'
ORDER BY created_at DESC
LIMIT 20;
```

- **План**: Последовательный скан на orders (100k строк) + Индекс скан на users
- **Execution Time**: 4.5 ms
- **Причина**: Полный просмотр таблицы orders, хотя пользователь мог иметь всего несколько заказов

### Запрос №2: Статистика по статусам заказов за период

```sql
SELECT status, COUNT(*) as order_count, SUM(total_amount) as total_revenue
FROM orders
WHERE created_at >= '2025-01-01' 
  AND created_at < '2025-04-01'
GROUP BY status
ORDER BY order_count DESC;
```

- **План**: Последовательный скан на orders (100k строк) + HashAggregate
- **Execution Time**: 6.9 ms
- **Причина**: Нет индекса по created_at, приходится сканировать все заказы

### Запрос №3: Топ-10 товаров по продажам

```sql
SELECT oi.product_name, 
       COUNT(DISTINCT o.id) as order_count,
       SUM(oi.quantity) as total_quantity,
       SUM(oi.price * oi.quantity) as total_revenue
FROM order_items oi
JOIN orders o ON o.id = oi.order_id
WHERE o.created_at >= '2025-01-01'
  AND o.created_at < '2025-04-01'
GROUP BY oi.product_name
ORDER BY total_revenue DESC
LIMIT 10;
```

- **План**: Последовательный скан на orders + Последовательный скан на order_items + Hash Join + GroupAggregate
- **Execution Time**: 150 ms
- **Причина**: Два полных скана больших таблиц и тяжёлая агрегация

### Запрос №4: "Зависшие" paid заказы
```sql
SELECT o.id, o.user_id, o.total_amount, o.created_at,
       (SELECT changed_at FROM order_status_history osh 
        WHERE osh.order_id = o.id AND osh.status = 'paid' 
        ORDER BY changed_at DESC LIMIT 1) as paid_at
FROM orders o
WHERE o.status = 'paid'
  AND EXISTS (
    SELECT 1 FROM order_status_history osh 
    WHERE osh.order_id = o.id AND osh.status = 'paid'
  )
  AND NOT EXISTS (
    SELECT 1 FROM order_status_history osh 
    WHERE osh.order_id = o.id AND osh.status = 'shipped'
  )
  AND o.created_at < NOW() - INTERVAL '1 day'
ORDER BY o.created_at;
```

- **План**: Последовательные сканы + подзапрос для каждой строки
- **Execution Time**: 231 617 ms (3 минуты 51 секунда)
- **Причина**: Подзапрос выполняется для каждого заказа (49k раз), нет индексов для быстрого поиска в истории

## 3. Добавленные индексы и обоснование типа

### Индекс №1

- **SQL**: `CREATE INDEX idx_orders_created_at ON orders USING BTREE (created_at);`
- **Какой запрос ускоряет**: Q1, Q2, Q3 - все запросы с фильтрацией по дате
- **Почему выбран тип**: BTREE оптимален для диапазонных запросов и сортировки по дате

### Индекс №2
- **SQL**: `CREATE INDEX idx_orders_status_created_at ON orders USING BTREE (status, created_at);`
- **Какой запрос ускоряет**: Q2 и Q4 (фильтрация по статусу + дата)
- **Почему выбран тип**: Составной BTREE позволяет сразу отсекать по статусу, а затем по дате

### Индекс №3
- **SQL**: `CREATE INDEX idx_order_status_history_order_id_status ON order_status_history USING BTREE (order_id, status, changed_at DESC);`
- **Какой запрос ускоряет**: Q4
- **Почему выбран тип**: Покрывающий индекс со всеми нужными полями, ORDER BY уже в индексе

### Индекс №4 (опциональный)
- **SQL**: `CREATE INDEX idx_orders_recent ON orders USING BTREE (created_at, id) WHERE created_at >= '2024-01-01';`
- **Какой запрос ускоряет**: Q3 (JOIN)
- **Почему выбран тип**: Частичный индекс для свежих данных - меньше размер, быстрее сканирование

## 5. Партиционирование `orders` по дате

### 5.1 Выбранная стратегия

Выбрана стратегия RANGE-партиционирования по кварталам (3 месяца). Это обеспечивает хороший баланс между количеством партиций и их размером.

### 5.2 Реализация

Создана демонстрационная партиционированная таблица `orders_partitioned_demo` с партициями по кварталам 2024-2026 годов. Данные скопированы из оригинальной таблицы. Добавлены индексы, аналогичные индексам на основной таблице.

### 5.3 Проверка эффекта

- **До**: Bitmap Index Scan по всей таблице (1073 buffer hits + 36 reads)
- **После**: Seq Scan только по одной партиции orders_demo_2025_q1 (133 buffer hits)
- Сокращение числа читаемых страниц в 8 раз.
- Время выполнения сократилось с 3.2 ms до 1.8 ms

## 7. Что удалось исправить

1. Катастрофически медленный Q4 - ускорен в 613× с помощью покрывающего индекса на order_status_history. Подзапрос теперь выполняется через Index Only Scan.
2. Фильтрация по дате - индексы на created_at позволили избежать полных сканов таблицы.
3. Фильтрация по статусу - составной индекс статус+дата ускорил группировку по статусам.
4. Партиционирование - сократило количество читаемых данных в 8 раз для запросов по диапазону дат.



## 8. Что не удалось исправить только индексами

1. Q3 (топ товаров) - остался медленным (около 150 ms) даже после всех оптимизаций. Индексы не помогли, потому что:

   - Нужно сканировать все order_items (400k строк)
   - Тяжёлая группировка и сортировка по product_name
   - Требуется JOIN с orders для фильтрации по дате
   
   *Решение*: потребуется либо материализованное представление с предрасчётом, либо изменение логики.

2. Q4 после партиционирования - стал немного медленнее, так как нужно сканировать несколько партиций вместо одной таблицы. Это плата за параллельное сканирование.

## 9. Выводы

1. Диагностика - ключ к оптимизации. EXPLAIN ANALYZE показал точные узкие места, позволил запланировать оптимизацию.

2. Покрывающий индекс, включающий все нужные поля, позволяет PostgreSQL не обращаться к таблице вообще (Index Only Scan).

3. Партиционирование эффективно для временных рядов. Для запросов с фильтром по дате partition pruning сокращает объём данных в разы.

4. Не все проблемы решаются индексами. Сложные агрегаты на больших таблицах (как в Q3) требуют архитектурных решений: предрасчёт, кэширование, отдельные таблицы статистики.

5. После индексов Q4 ускорился в 613 раз, но после партиционирования немного просел из-за особенностей реализации. Важно понимать, что в реальной системе нужно тестировать полный набор оптимизаций вместе.
